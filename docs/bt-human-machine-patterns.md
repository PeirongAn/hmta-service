# BT Human-Machine Collaboration Patterns

Three architectural approaches for representing human-machine collaboration in behavior trees.

## Approach A: Pure Decorator Mode

Every action is wrapped in a `HumanMachineDecorator` whose `mode` parameter
(set by the allocator's `bt_pattern`) determines runtime behavior.

### BT structure (LLM output)

```
selector[
    sequence[
        parallel(SuccessOnOne)[
            sequence[
                HM_decorator(mode="supervised_fallback")[ navigate(zone1) ]
                HM_decorator(mode="autonomous")[ scan(zone1) ]
                HM_decorator(mode="supervised_fallback")[ navigate(zone2) ]
                HM_decorator(mode="autonomous")[ scan(zone2) ]
                condition("bomb_detected", expected=true)
            ]
            sequence[ ... dog2 ... ]
        ]
        HM_decorator(mode="supervised_report")[ action("report", operator) ]
    ]
    sequence[ humanGate("path_planning"), follow_by_path ]
]
```

### Decorator modes

| mode | Runtime behavior |
|---|---|
| `autonomous` | Execute child directly; FAILURE = decorator FAILURE |
| `supervised_fallback` | Try child → on FAILURE dispatch humanGate(path_planning) → follow_by_path → retry up to N times |
| `supervised_report` | Execute child → on SUCCESS dispatch report to supervisor |
| `human_approve_execute` | Dispatch humanGate(approve) → on SUCCESS execute child |
| `human_plan_execute` | Dispatch humanGate(path_planning) → execute follow_by_path with operator waypoints |
| `human_remote_control` | Dispatch humanGate(remote_operate) — no child action |

### JSON schema

```json
{
  "node_id": "hm_nav_z1",
  "type": "hm_decorator",
  "name": "Navigate Zone 1 (supervised)",
  "mode": "supervised_fallback",
  "human_entity": "operator-01",
  "max_retries": 3,
  "children": ["a_nav_z1"]
}
```

### Pros

- Minimal BT nodes (~25 for bomb search scenario)
- LLM prompt is simple — just output action + mode
- All human-machine logic is in Python code, deterministic

### Cons

- Large runtime implementation (one decorator class handling 6 modes)
- Less visibility into BT structure (human fallback steps are hidden)
- Harder to debug — BT visualization only shows the decorator, not internal states

## Approach B: Hybrid Mode (SELECTED)

Core human-machine patterns use a single `supervised_fallback` decorator for
the navigate+scan zone visit pattern. Mission-level structure (parallel,
selector, condition) remains explicitly generated by the LLM.

### BT structure (LLM output)

```
selector[
    sequence[
        parallel(SuccessOnOne)[
            sequence[
                selector[ condition(cleared_z1), supervised_fallback[ seq[nav_z1, scan_z1] ] ]
                selector[ condition(cleared_z2), supervised_fallback[ seq[nav_z2, scan_z2] ] ]
                condition("bomb_detected", expected=true)
            ]
            sequence[ ... dog2 ... ]
        ]
        action("report", operator)
    ]
    sequence[ condition("all_zones_cleared"), humanGate, follow_by_path ]
    sequence[ humanGate, follow_by_path ]
]
```

### Decorator behavior

`supervised_fallback` wraps a child subtree (typically `sequence[navigate, scan]`):

```
State machine:
  AUTONOMOUS: tick child
    child SUCCESS → decorator SUCCESS
    child RUNNING → decorator RUNNING
    child FAILURE → save failure reason, transition to HUMAN_FALLBACK

  HUMAN_FALLBACK: dispatch humanGate(path_planning) directive
    poll for operator waypoints on Blackboard
    waypoints received → dispatch follow_by_path command
    follow_by_path SUCCESS → reset child, transition to AUTONOMOUS (re-tick child)
    follow_by_path FAILURE → retry (up to max_retries)
    all retries exhausted → decorator FAILURE
```

### JSON schema

```json
{
  "node_id": "sf_zone1",
  "type": "supervised_fallback",
  "name": "Navigate and Scan Zone1",
  "children": ["a_nav_z1", "a_scan_z1"],
  "human_entity": "operator-01",
  "max_retries": 3
}
```

### Pros

- Balanced: ~40 nodes (vs 72 explicit, vs 25 pure decorator)
- LLM still controls mission structure, easy to debug
- Only 1 decorator class to implement
- Zone guard (condition) remains explicit and visible

### Cons

- LLM must still generate condition/parallel/selector correctly
- Only covers the "navigate fallback" pattern — other modes still use explicit nodes

## Approach C: Explicit Node Mode (CURRENT)

All human-machine interaction patterns are generated as explicit BT nodes by
the LLM. No decorator abstraction.

### BT structure (LLM output)

```
selector[
    sequence[
        parallel(SuccessOnOne)[
            sequence[
                selector[
                    condition("zones/z1/cleared")
                    sequence[
                        selector[
                            action("navigation", dog1, zone_id="z1")
                            repeat(3)[ sequence[ humanGate("path_planning"), follow_by_path ] ]
                        ]
                        action("scan", dog1, zone_id="z1")
                    ]
                ]
                selector[ ... zone2 ... ]
                condition("bomb_detected", expected=true)
            ]
            sequence[ ... dog2 ... ]
        ]
        action("report", operator)
    ]
    sequence[ condition("all_zones_cleared"), humanGate, follow_by_path ]
    sequence[ humanGate, follow_by_path ]
]
```

### Pros

- No new Python code needed
- Full visibility — every node is in the BT visualization
- Maximum flexibility for LLM

### Cons

- 72+ nodes for bomb search scenario
- Complex LLM prompt (many rules for selector/repeat/humanGate patterns)
- LLM frequently generates incorrect structures (condition logic bugs, wrong intents)
- Duplicate logic across nodes (each zone has identical fallback scaffolding)

---

## Comparison

| Dimension | A (Pure Decorator) | B (Hybrid) | C (Explicit) |
|---|---|---|---|
| BT node count | ~25 | ~40 | ~72 |
| LLM prompt complexity | Low | Medium | High |
| Runtime code change | Large | Small | None |
| Human-machine control | Code-driven | Code+LLM | LLM-driven |
| Debuggability | Lower | Good | Highest |
| Error rate | Low | Medium | High |
| Recommended for | Production | MVP/Demo | Prototyping |
